# Conventions

## Language and UX

- User-facing README text is in Spanish.
- CLI help, progress messages, and error messages are also in Spanish.
- Internal memory files in `.opencode/` may be English or Spanish, but should remain concise and explicit.

## Architecture conventions

- The repo currently favors a single-script implementation over package decomposition.
- Related logic is grouped by stage inside `svg_to_paint_by_numbers_pdf.py` rather than by separate modules.
- If future refactors split the module, the maps in `.opencode/` must be reorganized accordingly.

## Rendering conventions

- Output is a single-page A4 PDF.
- The main artwork is rendered in grayscale, not in the original fill colors.
- Labels use one-character references only.
- The legend includes all detected colors and always includes pure black after normalization.
- Fallback label placement is preferred over silently dropping labels when geometry is too tight.

## File and folder conventions

- `inputs/` is the recommended source folder for batch work.
- `output/` is the default generated-output folder for single-file work.
- `pdf-output-*` folders are generated batch-output folders.
- `patterns/` stores pattern SVGs for mystery mode.
- `fonts/` stores required font assets.
- Generated outputs are not treated as source files.

## Documentation conventions

- `README.md` is the user-facing reference.
- `.opencode/project-map.md` is the structural repo map.
- `.opencode/python-map.md` is the implementation map for Python code.
- The `AST Inventory` section inside `.opencode/python-map.md` is machine-generated by `scripts/generate_python_map.py`.
- `.opencode/repo-state.md` is the operational snapshot for future prompts.
- Any change that alters behavior should update the relevant map, not just the code.

## Safety conventions

- Do not inspect or commit `.env` unless the user explicitly asks.
- Do not treat `__pycache__/`, generated PDFs, or historical logs as implementation files.
- Keep `.gitignore` and `.dockerignore` aligned with newly introduced generated or local-only artifacts.

## Style conventions inferred from the codebase

- Dataclasses are used for structured conversion state and results.
- Clear domain-specific exceptions are preferred over raw crashes for expected user errors.
- CLI feedback is rich and descriptive, especially for progress and validation errors.
- Performance-sensitive geometry flow uses fast prechecks such as bounds overlap and spatial indexing.

## Maintenance convention

Whenever a future prompt changes code, ask implicitly: what should the next agent need to know without rereading the whole repo?
If the answer changes, update `.opencode/` in the same edit.
